# Developer GTM Strategy Blueprint

## Purpose

Bridge product innovation to adoption through workflow-based strategy.

## Developer Lifecycle

1. **Reach** — create awareness and curiosity
2. **Engage** — deepen understanding
3. **Activate** — get the user to try a workflow
4. **Habit** — drive repeated use
5. **Expand** — increase depth and surface area of usage
6. **Advocate** — encourage sharing, templates, examples, and public proof

## Segments

### Fast Movers

Solo builders, startups, OSS maintainers, and power users.

Need: speed, agency, flexibility, advanced workflows.

### Enterprise Teams

Professional developers working inside larger organizations.

Need: reliability, clarity, safety, governance, ROI.

### Learners

Early-career developers, students, and structured learners.

Need: guidance, examples, scaffolded paths.

### Platform Owners

Admins, platform engineers, developer productivity teams, and enablement leaders.

Need: repeatability, control, governance, standardization.

## Core Insight

Adoption is repeated workflow success.

## Prioritization Rule

When in doubt, prioritize:

1. Workflows developers already understand
2. Workflows with visible proof artifacts
3. Workflows that create repeat usage
4. Workflows that connect multiple surfaces or teams

## Agent Instructions

Before creating any output, identify:
